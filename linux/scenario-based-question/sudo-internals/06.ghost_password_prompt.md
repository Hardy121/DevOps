# The Scenario 
You have a script that runs every hour via a user's `crontab`. The script contains a `sudo` command. You have already added the following line to `/etc/sudoers`:
`backup_user ALL=(ALL) NOPASSWD: /usr/bin/rsync`
When you run the script manually, it works perfectly. But when `cron` runs it, the script fails, and the logs show:
`sudo: a terminal is required to read the password; either use the -S option to read from standard input or configure an askpass helper`

# The Question 
There is a specific **Default** setting in the `sudoers` file that forces a user to have a "real" terminal (tty) to use sudo. What is that setting called, and how do you disable it for just this one user?

# The Answer
By default, many Linux distributions (like RHEL/CentOS) have a security setting in `/etc/sudoers` that says:
`Defaults requiretty`

This means `sudo` will **only** work if there is a real person at a real keyboard (a TTY). Since `cron` (the task scheduler) is a background daemon, it doesn't have a "terminal." When `sudo` sees no terminal, it panics and dies, even if you set `NOPASSWD`.

# The Fix:
You have to tell `sudo` to relax this requirement for that specific user. You would use `visudo` to add:

Code snippet

`Defaults:backup_user !requiretty`

The `!` negates the requirement. Now, `backup_user` can run `sudo` commands in the background (via cron or scripts) without the system demanding a physical terminal session.
