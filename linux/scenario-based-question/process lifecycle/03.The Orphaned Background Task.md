# The Scenario
The "Orphaned" Background Task
You are logged into a remote production server via SSH. You need to run a long-lived maintenance script, db_cleanup.sh, which takes 2 hours to complete.

Halfway through, your local internet connection drops, and your SSH session terminates. Usually, when an SSH session ends, the shell sends a SIGHUP (Signal Hang Up) to all its child processes, killing your script and potentially leaving your database in a messy, half-cleaned state.

# Your Task
Prevention: Before you even start the script, what command would you prefix it with to ensure it ignores the SIGHUP signal and keeps running even if you get disconnected?

Redirection: By default, where does the output (stdout/stderr) go when you use that command, and how can you point it to a specific log file instead?

Modern Alternative: In a modern DevOps workflow, many engineers prefer using a "terminal multiplexer" to handle this so they can re-attach to the session later. Can you name one of the two most popular tools used for this? (Hint: One starts with 's' and the other with 't').


# The Answer
1. `nohup ./db_cleanup.sh &`
2. `nohup ./db_cleanup.sh > cleanup.log 2>&1 &`
3. `tmux`
