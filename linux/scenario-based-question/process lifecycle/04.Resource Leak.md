# The Scenario:
You are managing a Kubernetes node (or a heavy-duty Linux VM). One of your Java-based microservices has a memory leak. It keeps consuming RAM until the system is almost out of memory.

Linux has a "bodyguard" called the OOM Killer (Out of Memory Killer) that jumps in to save the OS from crashing by killing a process.

# Your Task
Investigation: If your application suddenly disappears without you killing it, which log file (usually in /var/log/...) would you check to see if the kernel's OOM Killer terminated it?

Monitoring: Which command provides a real-time, dynamic view of processes sorted by Memory usage so you can spot the leak before the kernel kills it?

The "Score": Every process has an oom_score. If you have a critical process (like your SSH daemon) that you never want the kernel to kill, do you want its score to be very high or very low?

# The Answer
1. `dmesg`
2. `top or htop`
3.  The lower the score the less chance to be killed
