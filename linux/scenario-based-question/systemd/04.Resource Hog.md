# The Scenario 
Your API is now running and logs look clean. However, your monitoring tool (like Prometheus or Datadog) just sent an alert: The API service is consuming 95% of the server's RAM, causing other critical processes (like the database) to crash.

In a DevOps/SRE role, you can't just let one app take down the whole "node." You need to implement Resource Quotas directly into the systemd service file using Linux Control Groups (cgroups).

Your Goal:
Modify the api.service file to strictly limit the application so it can never use more than 500MB of RAM. If it tries to go over, the kernel should kill it (to protect the rest of the system).

# The Question
1. Inside the [Service] section, what is the name of the directive used to set a hard limit on memory usage (e.g., 500M)?
2. If you also wanted to ensure this API never takes up more than 20% of a CPU core, which directive would you add?
3. After you save these changes to the file, what are the two systemctl commands you must run (in order) to make the new limits take effect on the currently running process?

# The Answer    
1. `MemoryMax=500M`
2. `CPUQuota=20%`
3. `systemctl daemon-reload`
   `systemctl statys api.service`
