# The Scenario:
You have just deployed a custom Python-based Web API for your company's frontend. You placed the code in /opt/api/app.py.

The Problem: Currently, you are starting the API manually by running python3 /opt/api/app.py &. However, every time the server reboots for maintenance, the API doesn't come back up. Additionally, if the Python script crashes due to a memory leak or a bad request, it stays "dead" until someone manually logs in to restart it.

Your Goal:
As a DevOps engineer, you need to "service-ify" this application using systemd so it starts automatically on boot and restarts itself if it fails.

 # The Question
In which specific directory should you create the new service unit file?
What is the systemctl command you must run immediately after creating or editing that file to make sure systemd recognizes it?
Which section and directive inside the service file ensures the API starts automatically after a server reboot?

 # The Answer
1. file should be create inside `/etc/systemd/system/finename.serivce`
2. the commands should immediately run after creating systemd service : `systemctl daemon-reload`
3. `systemctl enable filename.service`
